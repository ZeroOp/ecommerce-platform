RedStore — System Architecture

RedStore is a microservices e-commerce platform. Every service is independently deployed on Kubernetes, owns its own database, and communicates through REST (synchronous) or NATS JetStream events (asynchronous). This document explains what every folder, file, and library is doing and why.

Services at a glance

ServiceLanguagePortWhat it does

identity-serviceJava8081Handles signup, login, JWT issuance, Google OAuth, seller registration, admin approval/rejection of sellers.
product-serviceJava8080Product catalog — brands, categories, product listings, metadata, image upload to MinIO via presigned URLs.
inventory-serviceJava8083Stock management — sellers add stock, orders reserve it, payment failure releases it. Redis caches quantities for fast public reads.
order-serviceJava8084Order lifecycle orchestration — creates orders, calls inventory reserve/release endpoints, tracks status transitions, and publishes order domain events.
payment-serviceJava8085Payment processing service — consumes order creation/expiration events, stores payment records, and publishes payment completion events for order progression.
analytics-serviceJava8086Read-optimized analytics service — consumes order lifecycle events and writes aggregates to ClickHouse for admin/seller dashboard reporting.
archive-serviceJava—Pulls every NATS event using durable subscriptions and writes them to MinIO as a permanent audit log.
snapshot-serviceNode.js—Cron job — runs pg_dump on Postgres databases every 2 hours and uploads the SQL files to MinIO for backup.
redstore-clientAngular 174200Single-page app — public store, seller dashboard, admin panel. Signal-based reactive state, OnPush change detection.

How a request travels through the system

Example: a seller adds stock to a product.

1
Browser → Angular
Seller types a quantity, clicks "Add stock". Angular's InventoryApiService fires a POST request.
2
Angular → Nginx Ingress
Request hits /api/inventory/seller/products/{id}/add-stock. Nginx Ingress matches /api/inventory/* and routes it to inventory-srv inside the cluster.
3
CurrentUserInterceptor (common)
Before the controller runs, the interceptor reads the JWT from the cookie or Authorization header. It parses it with JwtUtils and stores the user (id, roles) in a ThreadLocal called UserContext.
4
@RequireSeller (AOP in common)
RoleValidationAspect intercepts the method and checks UserContext for the SELLER role. If missing → HTTP 403 Forbidden, request dies here.
5
ProductOwnershipClient
inventory-service calls product-service's GET /api/seller/products/{id}, forwarding the original JWT. If the product doesn't belong to this seller → 400 Bad Request.
6
DB write + Redis cache
Upserts ProductInventory row in Postgres. Evicts old Redis key, writes new quantity with TTL so the public storefront reads it fast.
7
NATS JetStream publish
InventoryStockAddedPublisher serialises the event to JSON and publishes to the INVENTORY stream (inventory.stock.added). Other services (archive, order-saga) can consume it.
8
Response → browser
Returns updated {productId, sellerId, quantity}. Angular shows a toast notification and updates the UI signal.

Technology decisions

Spring Boot 3 Framework
Web framework for every Java service. Provides HTTP server (Tomcat), dependency injection, JPA integration, and auto-configuration.
Why: Removes boilerplate — no manual servlet config, connection pooling, or bean wiring.
PostgreSQL 16 Database
Primary relational database. Each service has its own dedicated Postgres cluster (via CloudNativePG operator in Kubernetes).
Why: Services are isolated — identity-db, product-db, inventory-db. One service's schema change can't break another.
NATS 2.10 + JetStream Events
Durable message bus. Publishers write events; consumers pull them at their own pace. Events survive restarts (stored on disk).
Why: Services don't need to be running at the same time. If archive-service is down, events queue up in NATS and are consumed when it comes back.
Redis 7 Cache
In-memory key-value store. Used in inventory-service to cache stock quantities so the public storefront doesn't hit Postgres on every page load.
Why: Stock reads happen far more often than writes. Redis keeps the DB load low at scale.
MinIO Object Storage
S3-compatible object store running inside the cluster. Stores product images, NATS event archives, and Postgres backup files.
Why: Product images shouldn't be stored in the DB. MinIO keeps them outside the application tier with presigned URLs for direct browser upload/download.
Kubernetes + Nginx Ingress Infra
Orchestrates all containers. Nginx Ingress is the single public entry point that routes requests to the right service by URL prefix.
Why: Centralised routing, TLS termination, and horizontal scaling without changing any service code.

Shared Library

Common Module

A Maven module (common/) that every Java service imports as a dependency. It holds zero business logic — only infrastructure code that would otherwise be copy-pasted into every service. No HTTP server, no database. Just shared utilities.

What problem does this solve?

Without this module, every service would have its own copy of JWT parsing, role checking, and exception handling — and they'd drift out of sync. The common module ensures that a role check in identity-service behaves identically to one in inventory-service.

Libraries used

JJWT 0.12.5 Auth
Java JWT library: jjwt-api, jjwt-impl, jjwt-jackson. Parses and validates JSON Web Tokens.
Why here: JWT parsing is needed in every service that protects endpoints. Centralised in common so all services use the same key and algorithm (HS256).
jnats 2.20.0 Events
Official Java NATS client. Used by BasePublisher to publish events to JetStream.
Why here: Every service publishes domain events. BasePublisher<T> in common handles JSON serialisation + publish so individual publishers only need to name the subject.
Spring AOP (AspectJ) Security
@Aspect, @Around — intercepts annotated methods at runtime.
Why here: RoleValidationAspect enforces @RequireSeller / @RequireAdmin automatically. Without this, every controller method would need a manual role check.
Spring Web MVC Interceptor
HandlerInterceptor — CurrentUserInterceptor runs before every controller method.
Why here: Parses the JWT once per request, stores the user in ThreadLocal<UserPayload> (UserContext), clears it after the response to prevent memory leaks.
Spring Validation Validation
Bean validation with annotations like @NotNull, @Min, @NotBlank.
Why here: Request DTOs in every service use these annotations. The GlobalExceptionHandler catches MethodArgumentNotValidException and returns a clean 400 JSON error.
Lombok Boilerplate
Code generation at compile time: @Getter, @Builder, @AllArgsConstructor, @Data.
Why: Entity and DTO classes need getters, setters, constructors, builders. Lombok generates all of these from annotations — no manual code.

Identity Service

The authentication authority of the platform. It is the only service that creates and signs JWTs. Every other service only reads the JWT — it never creates one. Also manages the seller lifecycle (request → admin review → approve/reject).

What does it accomplish?

User Authentication

Email/password login with bcrypt hashing. Issues a signed JWT with user id, email, and roles. Handles token refresh via refresh tokens stored server-side.

Google OAuth

Accepts Google ID tokens from the frontend, verifies them against Google's API, creates or retrieves the local user account, then issues a RedStore JWT.

Seller Registration

Any user can request a seller account. The request is stored with status PENDING. Admins approve or reject it through the admin API.

Event Publishing

Every important auth event (login, logout, seller approved, etc.) is published to the IDENTITY JetStream stream for the archive service to record.

Libraries used

spring-boot-starter-web HTTP
Embeds Tomcat, provides @RestController, @PostMapping, JSON serialisation via Jackson.
Why: The service exposes REST endpoints — signup, login, logout, currentuser, etc. This starter sets all of that up automatically.
spring-boot-starter-data-jpa Database
Spring Data JPA with Hibernate. Manages the User entity and provides UserRepository with methods like findByEmail().
Why: User accounts need to be persisted in Postgres. JPA handles SQL generation, connection pooling, and transactions automatically.
spring-boot-starter-data-redis Sessions
Redis client via Spring Data. Used for storing refresh tokens or session data.
Why: Refresh tokens need to be fast to look up and easy to invalidate (just delete the key). Redis is ideal for this.
JJWT 0.12.5 JWT
Signs and builds JWTs. Identity-service is the only place that creates JWTs using Jwts.builder().signWith(key).
Why: JWTs need to be signed with a secret key (JWT_KEY env var). JJWT is the standard Java library for this. Common module handles verification; identity-service handles creation.
google-api-client 2.5.0 OAuth
Google's official Java API client. GoogleTokenVerifier calls Google's tokeninfo endpoint to validate Google ID tokens from the frontend.
Why: You can't just decode a Google ID token on your own — you must call Google to verify it's genuine. This library handles that HTTP call and response parsing.
postgresql + spring-boot-starter-validation DB + Validation
JDBC driver for Postgres. Validation enables @Valid on request bodies so email/password format is checked before reaching service layer.
Why: Without validation, a blank password would reach bcrypt. Input validation happens as early as possible (controller level).

Product Service

The product catalog service. Manages brands, categories, and product listings. Sellers create products under approved brands; admins manage brands and categories. Product images are uploaded directly from the browser to MinIO using presigned URLs — the application server never handles image bytes.

What does it accomplish?

Product Catalog

Sellers create products with name, description, price, category, 3–6 images, and category-specific metadata (e.g. RAM/Storage for Mobiles). Products are only visible on the storefront if their brand is approved.

Brand Management

Sellers create brands. Admins approve or reject them. Only products under approved brands appear on the public storefront.

Category & Metadata

Admins define categories. Each category has a metadata template (JSON) that specifies required fields for products in that category. ProductService validates these on creation.

Image Upload (Presigned URLs)

Angular requests a presigned PUT URL from the product service. The browser then uploads directly to MinIO. The product service never receives the image bytes — only the resulting object key.

Libraries used

spring-boot-starter-web HTTP
REST API server. Handles all incoming HTTP requests for products, brands, categories.
Why: Core of the service — without this there's no HTTP server.
spring-boot-starter-data-jpa Database
Hibernate + Spring Data. Manages Product, Brand, Category entities in Postgres.
Why: Product data is relational — brands belong to sellers, categories link to brands (ManyToMany), products link to both. JPA handles these relationships cleanly.
software.amazon.awssdk:s3 v2.42 Storage
AWS SDK v2 S3 client. Pointed at MinIO (S3-compatible). Used to generate presigned PUT URLs (upload) and presigned GET URLs (read).
Why: Images can be large. Routing them through the app server wastes memory and CPU. Presigned URLs let the browser talk to MinIO directly. The SDK handles URL signing so MinIO accepts the upload.
spring-boot-starter-aop Security
Enables @Aspect support. Needed so RoleValidationAspect from the common module works in this service.
Why: AOP must be explicitly enabled in each service's Spring context. Without this starter, @RequireSeller annotations would be silently ignored.
spring-boot-starter-validation Validation
Bean validation for request bodies. Ensures product name is not blank, price is positive, image count is between 3 and 6, etc.
Why: Catches bad input at the controller layer before it reaches the service or database. Returns a clear 400 error via GlobalExceptionHandler.
h2 (test only) Testing
In-memory database. Used only during unit tests so tests don't need a running Postgres instance.
Why: Tests run faster and in isolation. No external database dependency during CI.

Inventory Service

Manages stock levels for every product. Sellers add stock. When a buyer places an order, the order-service atomically reserves the stock. If payment fails, the reservation is released and stock is restored. Quantities are cached in Redis so the public storefront never hits Postgres for reads.

What does it accomplish?

Stock Management

Sellers add units to their products. The ProductInventory table tracks how many units are physically on hand (after reservations are subtracted).

Reservation System (Saga)

The StockReservation table tracks which order holds which stock. The reserve operation is atomic — uses a SQL UPDATE WHERE quantity >= requested so two simultaneous orders can't both succeed if only one unit is left.

Redis Cache

Public stock reads go through Redis first. Cache miss → DB query + cache write. Stock changes always evict + rewrite the cache. TTL prevents stale data if a crash skips the cache update.

Internal API (API Key)

The internal endpoints (/api/inventory/internal/*) are protected by a shared secret header (X-Internal-Key) — not a JWT. These are called by the order-service machine-to-machine.

Libraries used

spring-boot-starter-data-redis Cache
StringRedisTemplate. Stores stock quantities as strings (e.g. "inventory:prd-123" → "42") with a configurable TTL.
Why: The product detail page and category page call /api/inventory/public/products/{id}/quantity for every product viewed. Without caching this would be thousands of DB hits per minute at scale. Redis makes it a microsecond in-memory lookup.
spring-boot-starter-data-jpa Database
JPA + Hibernate for ProductInventory and StockReservation entities.
Why: The atomic decrementIfSufficient() is a JPQL UPDATE query — this is the critical operation that prevents overselling. Only JPA/Hibernate lets you write this as a Java method and execute it as a single atomic SQL statement.
spring-boot-starter-aop Security
Enables AOP so RoleValidationAspect from common module works.
Why: Same as product-service — needed to activate the @RequireSeller aspect.
spring-boot-starter-validation Validation
Validates AddStockRequest: quantity must be @NotNull and @Min(1).
Why: Catches invalid input before it hits the service layer. A seller can't add 0 or negative units.
common (internal) Shared
JWT parsing, CurrentUserInterceptor, @RequireSeller, BasePublisher, exception handling.
Why: All shared infrastructure — no duplication between services.
spring-boot-starter-web (RestTemplate) HTTP Client
RestClientConfig creates a RestTemplate bean. Used by ProductOwnershipClient to call product-service.
Why: inventory-service needs to verify that a seller actually owns a product before letting them add stock. This is a synchronous HTTP call to product-service.

The oversell problem — why the atomic decrement matters:
If two requests for the last unit arrive simultaneously, a naive check-then-update would both read quantity=1, both decide "enough stock", both decrement to 0 — but the buyer would get 2 orders for 1 unit. The SQL UPDATE WHERE quantity >= requested runs atomically inside the database — only one of the two concurrent updates will match the WHERE clause.

Order Service

The order domain orchestrator. It creates customer orders, coordinates inventory reservation/release through internal APIs, advances order statuses, and publishes order lifecycle events for other services.

What does it accomplish?

Order Creation

Creates immutable order snapshots from checkout data and starts each order in a deterministic lifecycle state.

Inventory Coordination

Calls inventory internal APIs to reserve stock at order start and release stock on cancellation/failure paths.

Order Lifecycle

Transitions orders through states such as CREATED, IN_PROGRESS, SHIPPED, COMPLETED, FAILED, or EXPIRED based on business actions/events.

Event Publication

Publishes order domain events (created, in-progress, shipped, completed, cancelled/expired) to NATS JetStream for payment, analytics, and archive consumers.

Libraries used

spring-boot-starter-web HTTP
Exposes customer/seller/admin order APIs and internal endpoints used in orchestration flows.
Why: Order operations are user-facing and require synchronous responses for checkout and status views.
spring-boot-starter-data-jpa Database
Persists orders, line items, and status transitions in PostgreSQL.
Why: Order and line-item data must remain strongly consistent and auditable.
RestTemplate / HTTP client Internal Call
Calls inventory internal reserve/release APIs with shared internal key headers.
Why: Reservation status is required before confirming order creation.
common module Shared
JWT context, role validation, event DTOs, base publisher, and standard exceptions.
Why: Enforces consistent auth/event handling across services.

Payment Service

Responsible for payment record lifecycle and payment confirmation logic. It consumes order lifecycle events, validates payment actions, and emits payment-completion events for order progression.

What does it accomplish?

Payment Record Management

Creates and updates payment records per order to enforce one canonical payment state machine.

Order-Aware Validation

Uses order snapshots and order read clients to validate that payment is allowed for the target order state.

Event Consumption

Listens to order-created and order-expired events to initialize/close payment windows.

Event Publication

Publishes payment-order-complete events consumed by order service to move orders toward completion.

Libraries used

spring-boot-starter-web HTTP
Provides payment confirmation and payment detail endpoints.
Why: Checkout/payment confirmation needs immediate API response semantics.
spring-boot-starter-data-jpa Database
Stores `PaymentRecord` entities and transactional updates.
Why: Financially sensitive state requires durable and consistent writes.
NATS JetStream (jnats) Events
Subscribes to order events and publishes payment completion events.
Why: Decouples payment processing timing from order service availability.
common module Shared
Shared DTOs, subjects, and publisher abstractions.
Why: Guarantees consistent event contracts between order and payment domains.

Archive Service

A background worker with no HTTP API. It subscribes to every NATS JetStream stream using durable pull consumers. Every event published by any other service (user logins, product creations, inventory changes, orders) gets written to MinIO as a permanent JSON file — an immutable audit log.

What does it accomplish?

Why an audit log?

In a microservices system, individual services only know about their own domain. If something goes wrong (e.g. wrong inventory count, a disputed order), you need to reconstruct what happened across all services. The archive is a complete, chronological record of every event in the system — stored outside the databases so it survives even if a database is corrupted or dropped.

Libraries used

jnats 2.20.0 NATS
Java NATS client. The archive-service uses JetStream.subscribe() with durable pull subscriptions to consume events from four streams: IDENTITY, PRODUCTS, ORDERS, INVENTORY.
Why durable pull consumers: "Durable" means NATS remembers which messages this consumer has already seen. If archive-service restarts, it picks up exactly where it left off — no events are skipped or replayed twice.
software.amazon.awssdk:s3 v2.42 Storage
AWS SDK v2 S3 client. Pointed at MinIO. Writes each event as a JSON file to the redstore-events bucket.
Why MinIO/S3: Object storage is cheap, immutable by design, and unlimited in size. No schema migration needed when new event types are added — just write new JSON files.
spring-boot-starter (no web) Runtime
Uses the basic Spring Boot starter — not spring-boot-starter-web. This means no embedded HTTP server is started.
Why no web: The service has no API endpoints. It just runs subscription loops in background threads. Starting a Tomcat server would waste memory.
common (internal) Shared
Gives access to event DTO classes (e.g. InventoryStockAddedEventData) for deserialisation and the BasePublisher infrastructure.
Why: The event DTOs are defined once in common so publishers and consumers use the same class — changes only need to be made in one place.

How it works

1
NatsConfig creates 4 pull subscriptions on startup
Durable consumers named archiver-identity, archiver-products, archiver-orders, archiver-payments. Each subscribes to one subject pattern (identity.>, product.>, etc.).
2
Subscription creation retries
The createPullSubscription() helper retries up to 10 times with a 30-second delay. NATS cluster might not be ready immediately when the pod starts.
3
Background threads poll NATS for messages
Each subscription runs a loop: subscription.fetch(batch, timeout) → for each message: deserialise, write to MinIO, acknowledge the message.
4
MinIO write
File path pattern: redstore-events/{stream}/{subject}/{timestamp}_{messageId}.json. Event data written as pretty-printed JSON.
5
Acknowledge
After successful write, calls message.ack(). NATS marks this message as consumed. On restart, only unacknowledged messages are re-delivered.
What happens if MinIO is down? The message is not acknowledged, so it stays in NATS and will be reprocessed when MinIO comes back. NATS JetStream's durability means no events are permanently lost.
Node.js · Cron Job

Snapshot Service

A simple Node.js script that runs pg_dump on the Postgres databases every 2 hours and uploads the SQL dump files to MinIO. It is a database backup service — nothing more, nothing less.

What does it accomplish?

If a database needs to be restored (accidental deletion, corruption, rollback to an earlier state), you can download the latest .sql file from MinIO and run psql to restore it. Without backups, a database failure means permanent data loss.

Libraries used

node-cron Scheduler
Cron-style task scheduler for Node.js. Runs the snapshot task on a schedule defined by the SNAPSHOT_CRON_SCHEDULE environment variable (default: 0 */2 * * * = every 2 hours).
Why: Node doesn't have a built-in cron. This library translates a standard cron expression into a recurring timer. You can change the schedule by just changing an env var.
minio (Node SDK) Storage
MinIO JavaScript client. Used to create the backup bucket if it doesn't exist and upload the .sql file as an object.
Why: Uploads are streamed from the local filesystem to MinIO without loading the entire file into memory. Essential for large database dumps.
child_process (built-in) System
Node.js built-in module. Used to run pg_dump as a shell command via exec().
Why: There's no pure JS Postgres backup tool. The definitive way to backup Postgres is to run pg_dump — a C binary that ships with Postgres. Node calls it as a child process.
fs (built-in) File I/O
Node.js built-in filesystem module. Reads the .sql file from /tmp after pg_dump writes it, then streams it to MinIO. After upload, deletes the temp file.
Why delete after upload: The pod's /tmp disk is limited. Old dumps must be removed to prevent the disk from filling up.

How it works

// Simplified index.js flow:

// 1. On startup: run the snapshot task immediately (don't wait for first cron tick)
runSnapshotTask();

// 2. Schedule it on cron (default: every 2 hours)
cron.schedule(process.env.SNAPSHOT_CRON_SCHEDULE, runSnapshotTask, { timezone: 'UTC' });

// runSnapshotTask():
//   a. minio.makeBucket('redstore-main')  ← create if doesn't exist
//   b. For each database (product-db, identity-db):
//      i.  takePgDump() → runs: pg_dump -h {host} -U {user} {dbname} > /tmp/{timestamp}_{db}.sql
//      ii. uploadFileToMinio(filePath, destKey) → stream to MinIO under db-snapshots/
//      iii. fs.unlinkSync(filePath) → delete local temp file
Why not use Postgres's built-in backup? CloudNativePG (the K8s operator) can do continuous archiving via WAL — but that's complex to set up. This script is simpler: a full SQL dump every 2 hours is good enough for a development/early-production platform. You can restore to any 2-hour window.
Angular 17 · Standalone Components

Angular Client

A Single-Page Application (SPA). Three separate layout zones: the public store (anyone can browse), the seller dashboard (requires SELLER JWT), and the admin panel (requires ADMIN JWT). Uses Angular 17's new signal-based reactivity and OnPush change detection for performance.

What does it accomplish?

Public Storefront

Home page, category browsing, product detail with inventory check, cart, checkout, order history, user profile. Anyone can access without logging in.

Seller Dashboard

/seller/* routes. Products management (create with image upload), inventory management (add stock), brands, orders. Requires SELLER JWT to access.

Admin Panel

/admin/* routes. View all products (from all sellers) with real inventory counts, manage brands (approve/reject), categories, sellers, users. Requires ADMIN JWT.

API Communication

All API calls go through service classes. JWT is sent via cookie (HttpOnly) automatically by the browser, or as an Authorization header. The Angular dev proxy routes /api/* to the correct local service port.

Libraries used

@angular/core — Signals State
signal() creates reactive state. computed() derives values. The template calls the signal as a function: products().
Why signals instead of RxJS for local state: Signals are simpler — no Subjects, no BehaviorSubject, no subscribe/unsubscribe. They work perfectly with OnPush change detection because Angular knows exactly which components depend on which signals.
@angular/common/http (HttpClient) HTTP
Angular's built-in HTTP client. All API service classes inject HttpClient and return Observable<T>.
Why Observable not async/await: HttpClient integrates with Angular's interceptor pipeline. If you later add a global auth interceptor (to refresh the token), it applies everywhere automatically.
@angular/router Routing
Client-side routing. Lazy-loaded route guards (authGuard) check the JWT before allowing navigation to /seller/* or /admin/*.
Why route guards: Without guards, a user who knows the URL could navigate to /seller/products — they'd get 403 from the API, but the page would still partially render. Guards redirect unauthenticated users to the login page immediately.
@angular/forms (FormsModule) Forms
Used for two-way binding in forms like the inventory add-stock input ([(ngModel)]).
Why: The inventory page needs to track the "units to add" number per product row. ngModel binds the input value directly to a signal-managed state object.
ChangeDetectionStrategy.OnPush Performance
Applied to every component. Angular only re-renders a component when one of its input references changes or a signal it reads emits a new value.
Why: Without OnPush, Angular checks every component on every user interaction. With OnPush + signals, only the components whose data actually changed re-render — dramatically better performance on large lists.
RxJS (firstValueFrom) Async
Used in form submissions like product creation where you need to await a sequence of API calls (upload images one by one, then create product).
Why: firstValueFrom(observable) converts an RxJS Observable to a Promise so you can use async/await syntax in the component methods for linear async flows.

Skaffold dev loop

skaffold dev watches your files and automatically rebuilds and redeployes when you save.

# skaffold.yaml: each artifact has:
#   context: .          ← build context is the project ROOT (not the service folder)
#   dockerfile: services/{service}/Dockerfile

# Why root context?
# The Dockerfile copies common/pom.xml, services/, common/src/ —
# all paths relative to the project root, not the service directory.
# If you accidentally run:
#   docker build -f services/X/Dockerfile services/X   ← WRONG, will fail
# The correct command is always:
#   docker build -f services/X/Dockerfile .            ← correct

# Build stages:
# Stage 1 (maven:3.9.6):
#   1. Copy root pom.xml + all service pom.xml files
#   2. Run: mvn dependency:go-offline -pl services/{service} -am
#      ← Downloads all Maven deps. This layer is CACHED until pom.xml changes.
#   3. Copy source code (common/src, services/{service}/src)
#   4. Run: mvn clean package -DskipTests
#      ← Builds the JAR
# Stage 2 (eclipse-temurin:21-jre-alpine):
#   COPY --from=build *.jar app.jar
#   ← Final image is tiny: only JRE + the JAR, no Maven, no source code

Secret management

How secrets get into pods: app-secreates.yaml defines a Kubernetes Secret. Each Deployment's env: block references it with valueFrom.secretKeyRef. When the pod starts, Kubernetes injects those values as environment variables. Spring Boot reads them with @Value("${jwt.key}") or via application.properties using ${JWT_KEY} placeholders.
Never commit real secrets to git. The app-secreates.yaml file in the repo contains placeholder values. In production, real secrets are applied separately and not stored in version control.
